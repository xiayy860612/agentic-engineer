# 用户可以登录 admin-web

- Issue: https://github.com/xiayy860612/agentic-engineer/issues/3

## 背景与目标

为 **admin-web** 提供基于 **用户名 + 密码** 的登录能力，使用户能进入管理端受保护区域。认证真源为 **FastAPI HTTP API**（当前无现成后端，本需求包含 **新增 FastAPI 服务** 承担校验与登录相关契约）。

**目标**：在「已预先创建用户」的前提下，用户可在 admin-web 完成登录；对**未知用户**与**密码错误**采用**同一套**失败语义与界面提示（**不**区分「用户不存在」与「密码错误」），以降低用户名枚举风险并简化契约。

## 技术选型（可选）

| 项 | 结论 |
|----|------|
| 前端 | **admin-web**（Next.js 栈见 `admin-web/AGENTS.md`） |
| 后端 | **FastAPI**，对外提供登录等 HTTP API |
| 会话 / 令牌 | **本阶段不拍板**。在 **`plan.md`** 中给出唯一方案并在契约中固定（候选包括但不限于：服务端 Session + Cookie、JWT Bearer、JWT httpOnly Cookie 等） |
| 用户来源 | **不**在本需求内提供自助注册或自动造号；用户由脚本、DB 或运维流程**预先创建** |

**须在 `plan.md` 中落地**：API 基址与路径、CORS/Cookie 策略（若适用）、OpenAPI 或等价请求/响应类型、与 admin-web 的环境变量约定。

## 范围

1. **FastAPI**：实现用户名/密码校验、与 `plan.md` 一致的登录成功响应及错误响应形状；密码仅在服务端以安全方式处理（如强哈希存储，**禁止**明文落库或写日志）。
2. **admin-web**：登录界面（表单字段：用户名、密码）、调用登录 API、根据结果展示成功跳转或错误信息；成功后的**受保护区域**至少有一处占位（具体路由在 `plan.md` 写明）。
3. **契约**：登录请求为 JSON，至少包含 `username`、`password`（字段名可在 `plan.md`/OpenAPI 最终锁定，但须前后端一致）。
4. **凭据失败（泛化）**：对「未知用户名」与「错误密码」分别验收（见 **AC2**、**AC3**），但对外须返回**一致**的失败契约与 UI 提示，**禁止**通过响应字段或 HTTP 语义向客户端泄露二者差异。

## 非目标

- 用户自助**注册**、邀请、邮件/短信验证。
- **找回/重置密码**、修改密码（可作为后续 Story）。
- **MFA/2FA**、OAuth/SSO。
- 生产级部署、审计日志、风控与限流细则（除非 `plan.md` 明确纳入本迭代最小集）。
- 会话形态的最终选型（已显式推迟到 `plan.md`）。

## 验收标准（AC）

#### AC1 — 预置用户可登录

- **Given** 已通过带外方式在系统中创建用户 `U`，且密码为 `P`（与 `plan.md` 中说明的存储方式一致）  
- **When** 在 admin-web 登录页输入 `U` / `P` 并提交  
- **Then** HTTP 登录接口返回成功（具体状态码与成功载荷由 `plan.md`/OpenAPI 固定），且客户端进入 **`plan.md` 所定义的「已认证」可观测状态**（例如持有 Cookie、或取得 token 并按约定附加到后续请求等，**必须在 `plan.md` 中单选写死**）

#### AC2 — 未知用户名

- **Given** 用户名在系统中不存在  
- **When** 使用该用户名与任意密码调用登录接口（或于 admin-web 等价提交）  
- **Then**  
  - 响应为失败；失败契约为 **`plan.md` 固定的泛化凭据失败**（单一 HTTP 状态码 + 统一错误体形状；若含机器可读字段如 `error` / `code`，其取值**不得**专指「用户不存在」）。  
  - admin-web 展示与成功不同的**统一**用户可见错误文案（由 `plan.md` 或产品文案表固定，例如「用户名或密码错误」）。  
  - **与 AC3 对齐**：本场景下对客户端可观测的失败响应（状态码、约定字段集合及取值、界面文案）须与 **AC3 — 密码错误** 场景下**逐字段一致**，不得以任何对外差异暗示失败子类。

#### AC3 — 密码错误

- **Given** 用户已存在且密码已知  
- **When** 使用**错误**密码调用登录接口（或于 admin-web 等价提交）  
- **Then**  
  - 响应为失败；失败契约同样为 **`plan.md` 固定的泛化凭据失败**（与 AC2 **同一** HTTP 状态码与错误体形状；**不得**出现 `invalid_password` 等可单独标识「仅密码错误」、从而与「未知用户」场景相区分的对外取值）。  
  - admin-web 展示的用户可见错误文案与 **AC2** 场景**完全相同**。  
  - **与 AC2 对齐**：与 AC2 的 **Then** 中对客户端可观测部分一致；可通过同构断言或对比测试验证两场景不可区分。

#### AC4 — 配置与联调

- **Given** 本地按 `plan.md` 启动 FastAPI 与 admin-web  
- **When** 使用 `plan.md` 记载的测试用户创建方式完成带外建号  
- **Then** 可按 AC1–AC3 完成手工或通过自动化用例验证（自动化是否纳入本迭代由 `plan.md`/tasks 决定，但**契约须可测**）

## 假设与风险（可选）

**假设**

- 开发阶段 admin-web 与 FastAPI 可配置为互相可达（同源代理或 CORS 等由 `plan.md` 说明）。
- 带外创建用户的具体步骤写在 `plan.md` 或 `README` 片段中，避免「无法造数」阻塞验收。

**风险**

- **排障成本**：统一错误提示后，一线排障需依赖服务端日志或关联 ID（若引入），不得在对外 API 中恢复细分错误码。  
- **会话方案未决**：在 `plan.md` 定稿前，前后端集成易出现返工；应尽早锁定单一方案并更新 OpenAPI/类型。

## 待决项（显式）

以下须在 **`plan.md`**（及必要时 `tasks.md`）中关闭，本 `spec.md` 不替代其细节：

1. 登录 API 路径、版本前缀；成功与**泛化凭据失败**时的 HTTP 状态码与错误体（须同时满足 **AC2** 与 **AC3** 的「对外一致、不可区分子类」要求）。  
2. 会话维持机制（单选）及 admin-web 读取/转发方式。  
3. 密码哈希算法与参数、密钥与 `SECRET` 类环境变量管理（须符合根目录 `AGENTS.md` 秘密管理约定）。  
4. 受保护路由与未登录重定向规则。
