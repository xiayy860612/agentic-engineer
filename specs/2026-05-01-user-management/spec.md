# 用户管理

## 1. 概述

- **业务背景**：当前 auth-service 已有 User 模型（id, username, password_hash）和登录/登出/session 端点，但用户创建仅能通过 CLI 命令行操作（`uv run cli create-user`）。系统没有角色划分，也没有 Web 界面的用户管理能力。
- **问题陈述**：运维人员需要 SSH 到服务器执行 CLI 才能管理用户账号，效率低且不安全。管理员需要一个 Web 界面来增删改查用户，并通过角色区分控制权限。
- **成功标准**：超级管理员可在 admin-web 上完成用户列表查看、新增、编辑、删除、密码重置全流程，无需登录服务器执行 CLI。

## 2. 干系人与用户


| 干系人 | 角色 | 利益与关注点 |
|--------|------|-------------|
| 超级管理员（super_admin） | admin-web 唯一合法用户 | 需要通过 Web UI 管理所有用户账号 |
| 普通用户（user） | 无后台访问权限 | 无法登录 admin-web |
| 运维/DevOps | 系统维护者 | 数据库初始化 seed 超级管理员，后续管理由 Web 完成 |

## 3. 功能需求

### 3.1 角色模型

- **用户故事**：作为系统，我想要区分用户角色，以便控制 admin-web 的访问权限和用户管理能力
- **验收标准**：
  - 创建 `roles` 表，包含 `id`、`name`（唯一）、`description`
  - `users` 表新增 `role_id` 外键关联 `roles`
  - 数据库初始化时自动创建 `super_admin` 和 `user` 两个角色
  - 数据库初始化时自动 seed 一个超级管理员账号

### 3.2 登录权限控制

- **用户故事**：作为超级管理员，我想要通过 admin-web 登录进入管理后台，以便管理用户
- **验收标准**：
  - 登录时校验用户角色：仅 `super_admin` 角色可登录成功
  - `user` 角色调用 `/api/v1/auth/login` 返回 403，提示无权限
  - `/api/v1/auth/session` 返回当前用户角色信息（含 `role` 字段）

### 3.3 用户列表

- **用户故事**：作为超级管理员，我想要查看用户列表，以便了解所有用户账号状态
- **验收标准**：
  - 表格展示：用户名、角色名称、创建时间
  - 支持按用户名搜索
  - 支持分页
  - 仅超级管理员可调用该 API

### 3.4 新增用户

- **用户故事**：作为超级管理员，我想要创建新的用户账号（设定用户名、密码、角色），以便新管理员或普通用户获得账号
- **验收标准**：
  - 表单字段：用户名（必填）、密码（必填）、角色（必填，下拉选择）
  - 密码强度：最短 8 位
  - 保存成功后返回用户列表，新用户可见
  - 仅超级管理员可调用该 API

### 3.5 编辑用户

- **用户故事**：作为超级管理员，我想要修改用户信息（用户名、角色），以便维护账号信息
- **验收标准**：
  - 可修改用户名和角色
  - 超级管理员无法修改自己的角色
  - 保存后列表刷新
  - 仅超级管理员可调用该 API

### 3.6 删除用户（软删除/停用）

- **用户故事**：作为超级管理员，我想要删除/停用用户，以便阻止其继续使用系统
- **验收标准**：
  - 点击删除弹出二次确认弹窗
  - 软删除：`users` 表新增 `is_active` 字段，删除后设为 `false`
  - 被停用的用户无法登录（登录时校验 `is_active`）
  - 超级管理员无法删除自己
  - 仅超级管理员可调用该 API

### 3.7 重置密码

- **用户故事**：作为超级管理员，我想要重置任意用户的密码，以便在密码遗忘时恢复其访问
- **验收标准**：
  - 输入新密码（最短 8 位）
  - 提交后密码立即生效
  - 仅超级管理员可调用该 API

### 3.8 前端导航

- **用户故事**：作为超级管理员，我想要在导航栏看到"用户管理"菜单入口，以便快速进入管理页面
- **验收标准**：
  - Navbar 展示"用户管理"链接
  - 仅超级管理员登录后可见（可通过 session 中的 role 判断）

### 3.9 边界情况

| 场景 | 预期处理 |
|------|---------|
| 用户名重复 | 返回 409 Conflict 并提示"用户名已存在" |
| 删除最后一个超级管理员 | 阻止操作，返回 400 并提示"不能删除最后一个超级管理员" |
| 修改角色导致无超级管理员 | 阻止操作，返回 400 并提示"至少保留一个超级管理员" |
| 非超级管理员调用管理 API | 返回 403 Forbidden |
| 未登录调用管理 API | 返回 401 Unauthorized |
| 密码长度不足 8 位 | 返回 422 Validation Error |
| 用户不存在时编辑/删除/重置密码 | 返回 404 Not Found |

## 4. 非功能需求

- **性能**：用户列表分页 ≤20 条/页，响应时间 <500ms
- **安全**：
  - 密码使用 Argon2id 哈希（复用现有 `security.py`）
  - 用户管理 API 通过 session + role 双重校验
  - 禁止日志输出密码明文
- **可扩展性**：角色模型预留后续扩展空间（新角色可通过数据库或 API 添加）
- **可用性**：删除、修改自己角色等敏感操作有二次确认

## 5. 依赖与假设

- **依赖**：
  - 现有 auth-service 的 session 机制和密码哈希模块
  - 现有 admin-web 的 Navbar 和 shadcn/ui 组件库
- **假设**：
  - admin-web 仅超级管理员使用，不存在多租户场景
  - 密码强度仅要求最短长度，MVP 不做复杂度规则
  - 角色不由 Web 界面管理（数据库初始化或直接 SQL 即可）

## 6. 风险与开放问题

- **风险**：软删除后历史 session 仍然有效（直到过期），可能在 TTL 内继续访问
- **开放问题**：
  - 是否需要 session 强制失效机制（主动踢下线）？
  - 是否需要记录用户操作日志？
  - 是否需要用户头像/邮箱等扩展字段？

## 7. 范围外

- 用户自助修改密码/个人信息
- 角色本身的可视化管理（增删角色 UI）
- 用户自助注册/邀请流程
- 操作日志/审计
- 邮箱/手机号等扩展字段
- session 主动失效/踢下线
