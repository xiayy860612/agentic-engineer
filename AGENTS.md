# AGENTS

面向人类协作者与 AI 代理的仓库级约定：根目录本文件只写 **best practice**、**Project Context** 与质量门禁**原则**，**不写**具体 shell 命令。**在修改任一子项目前，必须阅读并遵循该子项目目录下的 `AGENTS.md`**（若尚无该文件，以该目录 `README.md` 为准，并应尽快补齐 `AGENTS.md`）。

---

## Core Principles（核心原则）

- **范围与最小改动**：只改完成任务所必需的文件；避免无关重构与「顺手清理」。
- **先读后写**：修改前阅读调用方与既有约定；命名、目录结构、导入风格与现有代码一致。
- **设计与架构（常用原则）**：遵循 **SOLID**（单一职责、开闭、里氏替换、接口隔离、依赖倒置）、**YAGNI**；在合适场景下兼顾 **DRY**、**KISS**，并与「范围与最小改动」一致，避免过早或过度抽象。
- **契约优先（前后端分离）**：接口或共享语义变更时，同步考虑契约、类型与消费者；禁止前端依赖未文档化或未约定的后端行为。
- **安全与秘密**：禁止将密钥、令牌、生产连接串提交入库；禁止在日志中输出敏感数据；环境变量与示例配置分离（细节见子项目文档）。
- **文档层级与协作**：先读本文件，再读目标子目录 `AGENTS.md`；冲突时以**更贴近改动路径**的子项目文档为准，但不得违背安全、最小改动与 **「测试先行」** 专章。交付说明须含变更摘要与**可复核的验证方式**（命令仅见子项目文档）。
- **跨端改动**：涉及 API 或共享类型时，对照 **Project Context**，并同时满足相关前端、后端子项目各自门禁与 **「测试先行」** 专章。
- **需求与设计变更管理**：实现过程中若发生需求或设计变更，必须**先**更新对应 `specs/<功能>/spec.md`、`plan.md`、`tasks.md`，经用户确认后，方可基于最新 spec 继续实现；禁止在文档未更新或未经确认的情况下擅自调整实现方向。

## 测试先行

针对**新行为**或**缺陷修复**：须先新增或调整**会失败**的测试（或本节表格中、且子项目认可的方式），再写实现使测试通过；禁止默认「先大段实现再补测」。默认 **red → green → refactor**。

**框架、目录与运行命令**由各子项目 `AGENTS.md` 定义。选型：**能单元测的优先单元测**；跨进程、跨端或持久化边界再补集成 / 契约 / E2E。

| 方式 | 适用场景与要求 |
|------|----------------|
| **单元测试** | 隔离依赖，覆盖领域逻辑、纯函数、分支与边界；单元测试覆盖率门禁见 **Quality gates**。 |
| **集成测试** | 多模块协作及与数据库、消息队列、外部 HTTP 等边界的真实或容器化联调。 |
| **契约测试（含消费者驱动契约）** | 多方约定请求/响应与版本；契约变更须**先**改契约与失败用例，再改实现与消费者。 |
| **组件 / UI 测试** | 组件行为、关键交互与可访问性（具体库由子项目定）。 |
| **端到端（E2E）测试** | 关键用户旅程；不替代应有的单元覆盖，避免慢、脆的堆砌。 |
| **快照测试** | 稳定输出对比；须人工审阅 diff，禁止无意义整批接受。 |
| **属性测试** | 复杂不变量或大量输入组合；在子项目栈支持时用于补强单元测试。 |

### 豁免

**纯文档**、**纯配置重排**、**仅重命名且无行为变化**须在 PR 或说明中**一句话声明理由**；其余情形不得豁免。覆盖率等其他门禁的例外仅按 **Quality gates** 办理。

## Project Context（项目上下文）

全仓库共享的产品与领域真值；**不含**工具版本与运行命令。

### 文档与规格目录（`docs` / `specs`）

| 目录 | 作用 |
|------|------|
| **`docs/`** | 沉淀**系统设计**与**核心领域模型**（及配套稳定说明）；长期维护，随实现演进更新。 |
| **`specs/`** | 按**每个功能**或 **User Story** 使用**独立子目录**，容纳该条目的**技术调研**、**方案设计**、**任务拆分**等过程产出。 |

**版本库策略**：**`specs/`** 已列入根目录 **`.gitignore`**，规格文件**不进入 Git**；贡献者在本地按下列约定维护目录与文件名即可，PR 中仍应引用对应 **Issue** 并简述范围。

每个 **`specs/<功能或 US>/`** 子目录中的文件约定如下：

| 文件 | 必须性 | 说明 |
|------|--------|------|
| **`spec.md`** | 必须 | 需求说明（含验收标准 AC 等）；须包含对应 **GitHub Issue** 的链接。 |
| **`research.md`** | 可选 | 方案调研、对比与决策依据。 |
| **`plan.md`** | 必须 | 解决方案（设计与实现路径）。 |
| **`tasks.md`** | 必须 | 基于 `plan.md` 的任务拆分与进度管理。 |

**归档与整合**：`specs` 中某功能 / User Story **完成后**，将其结论与必要背景**归档**，并**合并、吸收到 `docs/`**（避免 specs 长期堆积已交付且未再维护的草稿；具体归档步骤可由团队约定）。沉淀到 **`docs/`** 时，须把 **`spec.md` 中关联的 GitHub Issue 链接**一并写入 **`docs/`** 内对应该功能的章节（或等价结构），便于日后对照需求与讨论历史做跟踪。

## Git 与版本控制

- **分支**：自约定主干（如 `main`）检出；建议 `feat/`、`fix/`、`chore/` 等前缀。
- **提交信息**：推荐 [Conventional Commits](https://www.conventionalcommits.org/)；说明**为何**，完整可读。
- **拉取请求**：动机、主要变更、验证方式（可引用子项目 `AGENTS.md`）；大改动拆分。
- **勿提交**：构建产物、密钥、本地环境文件、依赖目录等；以 `.gitignore` 与子项目约定为准。

## Quality gates（质量门禁）

**原则**如下；命令、工具与 CI 任务名见各子项目 `AGENTS.md`。

- **合并 / 交付前**：对本次触及的每个子项目执行其文档中的 lint、typecheck、test 及团队要求的 build。
- **测试**：行为类变更须具备**可失败且已通过**的测试；评审可要求「先红后绿」证据。豁免仅 **「测试先行」** 专章所列情形，须在 PR **明示**。
- **单元测试覆盖率（强制）**：各子项目 **≥ 90%**，默认 **行覆盖率**；若改用分支覆盖率等，须在子项目 `AGENTS.md` **明示口径**。子项目须说明工具、本地与 CI 校验方式、经认可的**排除路径**。低于阈值合并须 PR 说明理由并经评审同意。

## 子项目文档约定

新建 `frontend/`、`backend/`、`apps/*` 等时，于该目录新增 `AGENTS.md`，至少包含：技术栈与目录说明；dev / lint / typecheck / test / build 的**命令**；测试与**覆盖率（门禁见根目录 *Quality gates*）**的跑法与 CI；环境变量与示例配置。

当前若无子项目目录，以上在创建目录时生效。
