# 产品需求审阅技能

本技能指导 AI 对产品需求与规格进行**结构化、高质量**的审阅。涵盖三种使用方式：

1. **审阅已有 PRD / 规格** — 从完整度、清晰度与可行性进行批评性分析  
2. **需求澄清访谈** — 通过结构化问题补齐缺失信息  
3. **改写 / 优化** — 将模糊需求改写为结构清晰、可执行的表述  

---

## 步骤 1：判定模式

阅读用户消息，判断适用哪种模式：

| 信号 | 模式 |
|---|---|
| 用户粘贴文档或规格 | 审阅模式 |
| 用户描述功能想法但没有成文文档 | 访谈模式 |
| 用户说「改进」「改写」「修正」等 | 改写模式 |
| 不明确 | 默认 **审阅 + 访谈** 组合 |

---

## 步骤 2：审阅模式 — PRD / 规格批评

审阅已有文档时，从以下维度评估：

### 2.1 完整度清单
逐项标注 ✅ 已有 / ⚠️ 部分 / ❌ 缺失：

- **问题陈述**：用户问题是否说清？  
- **目标用户**：是否定义画像或用户细分？  
- **目标与成功指标**：是否有可衡量的 KPI 或验收标准？  
- **范围**：是否明确「做」与「不做」？  
- **用户故事 / 用例**：主路径与边界场景是否覆盖？  
- **功能需求**：是否用具体、可测试的方式描述？  
- **非功能需求**：性能、安全、可扩展、无障碍等？  
- **依赖**：上下游系统、API、团队？  
- **假设与风险**：未知与风险是否记录？  
- **时间线 / 里程碑**：是否有交付计划？  

### 2.2 清晰度问题
标出存在以下问题的需求：

- **模糊**：如「要快」「易用」「可扩展」— 无量化定义  
- **歧义**：可作多种理解  
- **矛盾**：两条需求互相冲突  
- **被动语态 / 缺主语**：如「数据应被保存」— 谁保存？何时？  

### 2.3 可行性标记（若有技术上下文）
- 可能与已知系统约束冲突的需求  
- 相对既定时间线明显过大的范围  
- 缺失的技术依赖  

### 2.4 审阅输出格式

按如下结构呈现结论：

```
## PRD 审阅摘要

### 完整度评分：X/10

| 维度 | 状态 | 说明 |
|---|---|---|
| 问题陈述 | ✅ | ... |
| 成功指标 | ❌ | ... |
...

### 优先处理的问题
1. [严重] ...
2. [重要] ...
3. [可择机] ...

### 澄清问题
- ...

### 建议改写（针对选中的模糊条目）
> 原文：「...」
> 建议：「...」
```

## 步骤 3：访谈模式 — 需求澄清

用户仅有想法、无成文文档时，进行结构化访谈。

每批最多 **3～4 个问题**，避免信息过载。根据回答调整后续追问。

### 第一轮 — 核心问题
- 要解决什么问题？谁遇到这个问题？  
- 上线约三个月后，「成功」长什么样？  
- 本功能**必须**做到的最重要的一件事是什么？  

### 第二轮 — 用户与流程
- 主要用户是谁？（角色、场景、技术熟练度）  
- 请走一遍关键用户旅程 — 用户逐步做什么？  
- 你最担心的常见失败或边界场景是什么？  

### 第三轮 — 约束与风险
- 硬约束有哪些？（截止日、平台、合规、预算等）  
- **明确不做**的是什么？（范围边界）  
- 你依赖哪些可能被证伪的假设？  

### 第四轮 — 完成定义
- 工程师如何判断某条需求已「完成」？  
- 首发的验收标准是什么？  
- 谁需要签字确认？  

每轮结束后，**先小结已了解内容**，再继续下一轮。

## 步骤 4：改写模式 — 改进需求

被要求改写时，套用以下模式：

### 坏例子 → 好例子

| 反模式 | 修正方向 |
|---|---|
| 「要快」 | 「在 4G 网络下，页面加载 95 分位 ≤ 2 秒」 |
| 「界面要友好」 | 「新用户无需帮助文档，3 分钟内完成引导」 |
| 「支持大文件」 | 「支持上传至 500MB；显示进度条；失败可恢复或明确提示」 |
| 「管理员可管理用户」 | 「管理员可邀请、停用用户，并在其组织内重新分配角色」 |

### 用户故事模板
```
作为 [角色]，
我想要 [行为]，
以便 [价值 / 结果]。

验收标准：
- 在 [上下文] 下，当 [操作] 时，则 [结果]
- 在 [上下文] 下，当 [操作] 时，则 [结果]
```

（可与英文习惯写法 As a / I want / So that 及 Given-When-Then 混用，以团队约定为准。）

## 语气与风格

- **直接且建设性** — 问题说清，并始终给出可操作的改进建议  
- 问题分级：严重（阻碍上线）→ 重要（影响质量）→ 可择机  
- 贴近用户用语（产品话术 / 技术 / 创业团队口语）  
- 若需求本身写得很好，应明确肯定 — 不要为凑篇幅硬找茬  

---

## 边界情况

- **极短规格（如「我们要一个登录页」）**：直接进入访谈模式  
- **很长文档（如 10+ 页）**：归纳**前 5 个**重点问题，而非穷举清单  
- **非中文输入的需求**：使用与输入**相同语言**回复  
- **仅有敏捷用户故事（无完整 PRD）**：审阅侧重 **INVEST** 标准（Independent 独立、Negotiable 可协商、Valuable 有价值、Estimable 可估算、Small 足够小、Testable 可测试）
