# 从 Spec 拆任务、实现

在**仓库根目录**执行。须已安装并登录 **GitHub CLI**（`gh auth status`；未登录则提示 `gh auth login`）。目标目录须含 **`spec.md`** 与 **`plan.md`**（约定见根目录 [AGENTS.md](../../AGENTS.md) 的 `specs/` 说明）。**实现与提交须在相对 `main` 等默认分支新检出的特性分支上进行**（§0.5）。

## 流程概览

| 阶段 | 内容 |
|------|------|
| §0 | 用户**必填** spec 目录路径；校验存在性与 `spec.md` / `plan.md` |
| §0.5 | **Git**：须在相对默认分支**新检出**的特性分支上执行（见下）；**禁止**在 `main` 等默认分支上提交本流程产生的实现 |
| §1 | 读真值；**生成或同步** `tasks.md`（续跑保留 `- [x]`） |
| §2 | 自首个 `- [ ]` 起实现；子项目门禁通过后再勾 `[x]` |
| §2.5 | 实现项全部完成后：将 **`spec.md` / `plan.md`** 中应长期保留的内容**吸收进 `docs/`**（见下；与根 [AGENTS.md](../../AGENTS.md)「归档与整合」一致） |

实现完成后的**提交、推送与开 PR**，请使用 **`spec-commit`** 命令。

用户若声明**仅**执行某一阶段（例如只同步 `tasks.md`、**本轮不更新 `docs/`**），以用户当期指令为准，跳过其后阶段；跳过 `docs/` 时须在回复中**写明**。

---

## 0. Spec 目录（必填）

用户须在触发时**显式给出**路径，例如：

- 相对仓库根：`specs/1-搭建前后端基础框架`
- 绝对路径：`/path/to/level-up/specs/1-搭建前后端基础框架`

**禁止**：未获路径时从对话推断、默认最近打开的 spec、或在多个 `specs/*/` 中自行挑选。

- **未提供路径**：停止；仅请用户补路径（并说明该目录下须有 `spec.md` 与 `plan.md`）。
- **路径无效或缺文件**：说明缺什么；不生成 `tasks.md`、不改实现代码。

校验通过后，**若存在**同目录 **`tasks.md`**，读入以供 §1 判定续跑。

---

## 0.5 Git：必须在特性分支执行（必填）

**硬性要求**：凡会修改**已纳入 Git 的代码或配置**并进入 **§2** 的步骤，**必须**在相对团队**默认分支**（一般为 `main`，以 `origin/HEAD` 或仓库约定为准）**单独检出**的**特性分支**上进行；**禁止**在默认分支上直接开发、提交本 spec 对应实现。

1. **开始前**（进入 §1 之前，或与 §1 并行最先执行）：`git branch --show-current`（或 `git status -sb`）确认当前分支名。
2. **若当前为默认分支**（如 `main` / `master`）：**停止**改代码与提交；先与远端同步默认分支（如 `git fetch origin && git checkout main && git pull`），再执行 **`git checkout -b <新分支>`**。分支名建议带前缀并与 Issue/spec 呼应，例如 `feat/1-scaffold`、`feat/<简述>`（Conventional 风格）。
3. **续跑同一 spec**：应在**此前为该工作创建的同一特性分支**上继续（`git checkout <该分支>`）；**不要**在默认分支上续跑提交，也**不要**无故换名另开分支导致同一 PR 分裂（除非用户明确要求重开分支）。
4. **仅 §1、且只改本地被 ignore 的 `tasks.md`**、无任何已跟踪文件变更时：可不切换分支；但一旦产生对仓库内已跟踪文件的修改或即将提交，**必须**已处于符合上文的特性分支。

---

## 1. 读取真值并生成 / 同步 `tasks.md`

1. **阅读** 该目录下 **`spec.md`**、**`plan.md`** 全文。
2. **提取 Issue**：从 `spec.md` 顶部 Issue 行或 `plan.md` 中对应行得到 **URL** 与 **编号**。不一致时以 **`spec.md` 为准** 并提醒核对。
3. **按 `tasks.md` 现状分支**（互斥，取最先匹配）：

   | 条件 | 动作 |
   |------|------|
   | **A. 无 `tasks.md`，或仅有空壳/无 `- [ ]`/`- [x]` 任务行** | **首次拆分**：写满任务列表（见下「任务体规范」） |
   | **B. 有未完成项**（至少一行 `- [ ]`） | **续跑**：**不得**清空列表或把 `- [x]` 改回 `- [ ]`；对照 `spec.md`/`plan.md` **合并更新**（增改条目、删除与真值矛盾且未开始的项）；**已 `[x]` 默认保留**，与新版真值冲突时先向用户说明再改 |
   | **C. 全部为 `- [x]`** | **不跑** §2；提示用户可用 **`spec-commit`** 提交并开 PR。用户**仅想重拆任务**时，须**明确说出**后再改 `tasks.md` |

4. **任务体规范**（首次拆分或合并更新时遵守）：

   - 一级标题示例：`任务拆分与进度 — <与 spec 标题一致或简写>`。
   - 文首列表：链到同目录 `spec.md`、`plan.md`、Issue。
   - 条目与 **`plan.md` 章节顺序**一致，可分子章节（前端 / 后端 / CI / 收尾等）；每条 **`- [ ]`** 或 **`- [x]`**，表述可验收；可标注 **(ACn)**。
   - **禁止**臆造 `spec.md` / `plan.md` 未出现的范围。

5. **统计**（用于回复）：只计任务清单里的 **`- [ ]` / `- [x]`** 行（不含文首元信息列表）。

**§1 结束回复**：`tasks.md` 路径、总任务数、已完成数 / 未完成数；续跑时加一句**将从哪一条（或哪一节）继续**。

---

## 2. 按 `tasks.md` 逐步实现

**开始前**满足 §0.5（已处于特性分支）。自 **第一个 `- [ ]`** 起顺序执行（**跳过** 已为 `- [x]` 的条目，除非用户要求回滚）。并行度由 `tasks.md` 语义决定，可说明「本节可并行」。

1. **先读** 将改动所涉子项目的 **`AGENTS.md`**；遵守根目录 [AGENTS.md](../../AGENTS.md)：**测试先行**、覆盖率、**最小改动**。
2. **新行为或缺陷修复**：**red → green → refactor**；先加/改**会失败**的测试再写实现（子项目另有规定从其规定）。
3. **每完成一条**：跑该子项目文档中的 **lint、typecheck、test**（及本次触及的 **build**）；**未通过不得**勾 `[x]` 或宣称完成。
4. 通过后将该条改为 **`- [x]`** 并保存 `tasks.md`。

**全部 `[x]` 后**：对本次触及的所有子项目再跑一遍门禁确认无遗漏。

---

## 2.5 吸收至 `docs/`（实现完毕后）

**目的**：本地 **`spec.md` / `plan.md`** 若不进入 Git（见根 `.gitignore` 策略），仓库内**权威叙述**仍须在 **`docs/`** 中可追溯；并与根目录 [AGENTS.md](../../AGENTS.md) **「归档与整合」**一致：功能交付完成后，将结论与必要背景**合并、吸收到 `docs/`**，且**写入对应 GitHub Issue 链接**。

**时机**：`tasks.md` 中与**实现及验收**相关的条目均已 **`[x]`** 后执行（除非用户当轮声明**跳过 `docs/`**）。

**须完成**：

1. **重读**当前 spec 目录下 **`spec.md`**、**`plan.md`**（若存在 **`research.md`** 且含有效决策，一并参考）。
2. **查看**仓库 **`docs/`** 现有结构（索引、`README`、分主题文件等）；若无合适落点，在**不臆造整套体系**的前提下新增最小合理文件（如 `docs/<主题>.md` 或在既有总览中增章），并与团队已有风格一致。
3. **写入 / 更新**（提炼整合，**非**全文粘贴）：例如背景与目标、范围与非目标、关键 **AC** 摘要、架构与目录边界、API/OpenAPI 与契约约定、已锁定技术选型、响应式/验收方式等——以**与当前代码、子项目 `AGENTS.md` 一致**为准；过时表述删除或改写。
4. **Issue 链接**：在 `docs/` 中对应章节（或文首元数据）**显式保留** `spec.md` 中的 **Issue URL**，便于对照需求与讨论历史。
5. **`git diff docs/`** 自检：增量可读、链接可点、无敏感信息。

**禁止**：把 `plan.md` 的任务勾选或过程草稿原样堆进 `docs/`；应用维护者视角**压缩为长期真值**。

---

## 实现完成后

实现与 `docs/` 吸收均完成后，使用 **`spec-commit`** 命令进行提交、推送并创建 / 更新 PR。

**§2 / §2.5 结束回复清单**：

- **当前 Git 分支名**（须为特性分支，§0.5）
- `tasks.md` 路径；是否**全部** `[x]`
- **`docs/`**：新增或修改的文件路径；若跳过 §2.5，说明原因
- 提示用户可执行 `spec-commit` 完成提交与 PR 流程
