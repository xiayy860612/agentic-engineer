# 从 Spec 工作树提交、推送并开 PR

在**仓库根目录**执行。须已安装并登录 **GitHub CLI**（`gh auth status`；未登录则提示 `gh auth login`）。本命令负责将已完成的 spec 实现提交到特性分支、推送到远端并创建或更新 PR。

## 流程概览

| 阶段 | 内容 |
|------|------|
| §0 | 校验：须在特性分支上（非默认分支）；确认变更与本次 spec 相关 |
| §1 | 提交代码 |
| §2 | 推送并创建 / 更新 PR（关联 Issue） |
| §3 | 失败处理与回滚 |

用户若声明**仅**执行某一阶段（例如只提交不开 PR、只推送不提交），以用户当期指令为准，跳过其后阶段。

---

## 0. 前置校验

1. **Git 分支检查**：`git branch --show-current`（或 `git status -sb`）确认当前分支名。
   - **若当前为默认分支**（如 `main` / `master`）：**停止**；提示用户先检出或创建特性分支（如 `git checkout -b feat/<简述>`）。**禁止**在默认分支上提交。
   - **若当前为特性分支**：继续。
2. **变更范围**：`git status` 确认变更应**仅**与本次 spec 相关（含 `docs/` 吸收内容）；勿纳入构建产物、依赖目录、本地环境文件；常见误加项参考根与子项目 **`.gitignore`**。

校验不通过时停止并说明原因；不提交。

---

## 1. 提交代码

1. **再次确认**当前分支**不是**默认分支（§0）。
2. **Conventional Commits**；说明**为何**；可一条或多条提交。
3. 在仓库根执行 `git add` / `git commit`（由执行环境完成）。

---

## 2. 推送并创建 PR（关联 Issue）

用户须提供 **Issue 编号**（或在 spec 目录路径下从 `spec.md` 顶部提取）。

1. **推送**当前分支至 `origin`（新分支设 upstream）。
2. **避免重复 PR**：推送后可用 `gh pr status` 或 `gh pr list --head <当前分支>`；若已有 PR，用 **`gh pr edit`**（必要时补 `Fixes #n`）更新描述，**不要**再 `gh pr create` 一条重复项。
3. **新建 PR** 时用 **`gh pr create`**：
   - **标题**：反映本次交付（可对齐 spec 标题）。
   - **正文**须含：**摘要**、**验证方式**（实际跑过的命令或引用 `AGENTS.md` + 已执行说明）、**需求追溯**、**`docs/` 变更摘要**（更新了哪些路径、或写明「已与现有 docs 核对无增量」）、单独一行的 **`Fixes #<编号>`** / **`Closes #<编号>`** / **`Refs #<编号>`**。
   - **需求追溯**：若 `specs/` **已纳入 Git**，在正文中给仓库内 `spec.md` / `plan.md` / `tasks.md` 的链接；若根 `.gitignore` **忽略 `specs/`**（本仓库当前策略），则改为 **Issue 链接** + 说明「规格与任务见本地 `specs/<目录>/`」，**勿**链接仓库中不存在的路径。
   - **Base 分支**按团队约定（多为 `main`）。
4. 捕获输出的 **PR URL** 回复用户。

---

## 3. 失败与回滚

- **`gh` 失败**：贴完整错误；不伪造 PR 链接。
- **Lint / test / build 失败**：先修复；PR 描述不得虚假声称已通过。
- **误在默认分支上已提交但未推送**：与用户确认后，可用新分支 + `cherry-pick` 或 `reset`/`revert` 等搬迁，**禁止**在未经确认时强推改写共享默认分支历史。
- **用户中途要求停止**：保存已提交进度；下次用**同一特性分支**续跑即可。

---

## 完成后回复清单

- **当前 Git 分支名**（须为特性分支）
- 提交摘要或 commit 范围
- **PR URL**（若本轮未开 PR，说明原因）
- Issue 编号与 PR 中使用的关键词（`Fixes` / `Closes` / `Refs`）
